# 作業計画

## MVP定義

**「ローカルでターミナルを起動してWSL2に接続してコマンドを打てる（テスト済み）」**

MVPは Phase 0 (開発基盤) + Phase 1 (基本ターミナル) の完了を指す。

---

## フェーズ構成

| Phase | 名称 | 概要 | MVP |
|---|---|---|---|
| 0 | 開発基盤構築 | プロジェクト初期化、ツールチェーン、CI/CD | Yes |
| 1 | 基本ターミナル | xterm.js + PTY + WSL2接続 | Yes |
| 2 | タブ管理 | 複数タブの作成・切替・閉じる | No |
| 3 | ペイン管理 | tmuxライクなスプリット・リサイズ | No |
| 4 | Claude Code統合 | セッションリスト・状態監視・通知 | No |
| 5 | 内蔵ブラウザ | ブラウザパネル・ターミナル連携 | No |
| 6 | セッション永続化 | プロセス維持・状態復元 | No |
| 7 | 設定・カスタマイズ | プロファイル・キーバインド・テーマ | No |

### T-1.3: Tauriコマンド (IPC) 実装

**実装内容**:
- PTY操作のTauriコマンドを定義
  - `create_pty(shell: String) -> PtyId`: PTY生成
  - `write_pty(id: PtyId, data: String)`: PTYへの入力
  - `resize_pty(id: PtyId, cols: u16, rows: u16)`: リサイズ
  - `close_pty(id: PtyId)`: PTY終了
- PTY出力のイベントストリーム (`pty-output` Tauriイベント)

**受け入れ条件**:

| # | 条件 | 検証方法 |
|---|---|---|
| A1 | `create_pty` コマンドがPtyIdを返す | 機械: `tauri::test` mockランタイムでコマンドを呼び出しIDが返ることをテスト |
| A2 | `write_pty` でデータが書き込まれる | 機械: 統合テストで書き込み後にPTY出力イベントが発火することをテスト |
| A3 | `resize_pty` がエラーなく完了する | 機械: 統合テストでリサイズコマンドが成功することをテスト |
| A4 | `close_pty` でPTYが終了する | 機械: close後に再度write/resizeするとエラーが返ることをテスト |
| A5 | PTY出力が `pty-output` イベントとしてフロントエンドに届く | 機械: フロントエンドのmockIPCテストでイベント受信をテスト |

### T-1.4: フロントエンド-バックエンド接続

**実装内容**:
- `TerminalView`コンポーネントからTauriコマンドを呼び出し、xterm.jsとPTYを双方向接続
  - コンポーネントマウント時に `create_pty` → PTY生成
  - xterm.js `onData` → `write_pty` → PTYへ入力転送
  - `pty-output` イベント → xterm.js `write()` → 画面出力
  - xterm.js `onResize` → `resize_pty` → PTYリサイズ
  - コンポーネントアンマウント時に `close_pty` → PTY終了
- Zustandストアでターミナル状態を管理

**受け入れ条件**:

| # | 条件 | 検証方法 |
|---|---|---|
| A1 | アプリ起動でターミナルが自動的に接続される | 人間: アプリ起動後にシェルプロンプトが表示されることを確認 |
| A2 | キー入力がPTYに送信され結果が表示される | 人間: `echo hello` と入力してEnterを押し `hello` が表示されることを確認 |
| A3 | ターミナルリサイズが正常に動作する | 人間: ウィンドウリサイズ後にコマンド出力のレイアウトが崩れないことを確認 |
| A4 | mockIPCテストが存在する | 機械: VitestでmockIPCを使ったIPC接続テストがpassする |
| A5 | Zustandストアの状態遷移テストが存在する | 機械: ストアの状態管理ロジックのユニットテストがpassする |
