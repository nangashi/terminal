# 設計原則（Design Principles）

## 目的

このドキュメントはプロジェクト全体の設計判断を導く原則を定義する。
個別のルール（`.claude/rules/`）やレイヤー境界設計書（`docs/layer-boundary.md`）の上位に位置し、
ルールが存在しない状況での判断基準として機能する。

---

## 原則一覧

### P1: ロジックはRustに寄せる

**判断ロジック・データ変換・バリデーションはRust側に置く。フロントエンドは「表示」と「ユーザー入力の受け取り」に専念する。**

TypeScript側に書いた方が楽な場面は多いが、以下の理由からRust側に集約する:

- Rustの型システムとコンパイル時チェックにより、ロジックの正しさを静的に保証できる
- Domain層のテストはPTYの実機テストで完結し、UIのモック不要で高速・安定
- フロントエンドにロジックが漏れると、Rust側との二重管理が発生する

#### 判断基準

「この処理はターミナルのUIが無くても意味があるか？」→ Yes なら Rust側に置く。

#### 具体例

| 処理 | 配置先 | 理由 |
|---|---|---|
| シェルの選定（OS判定） | Rust (Domain) | UI非依存のプラットフォームロジック |
| OSC 7パース（CWD抽出） | Rust (Domain) | PTY出力のドメイン知識 |
| Git情報の取得 | Rust (Domain) | ファイルシステム操作 |
| タブの並び順管理 | TypeScript (Hooks/App) | 純粋にUI状態 |
| ペイン分割のレイアウト計算 | TypeScript (Lib) | UI表示ロジック（React非依存なのでLib） |

**関連ルール**: `layer-boundary`

### P2: 境界を越えるデータは型と定数で守る

**IPC境界・モジュール境界で交換するデータは、型定義と定数で明示的に契約する。暗黙的な文字列ベースの契約を排除する。**

Rust↔TypeScript間にはコンパイル時の型チェックが効かない。この境界での不整合はサイレントに障害を引き起こす（イベントが届かない、データ構造が不一致等）。

#### 守るべきこと

- **イベント名・コマンド名**: 両言語で定数として定義する（typoを防ぐ）
- **データ構造**: Rust側の`#[derive(Serialize)]`型とTypeScript側の型定義を一致させる
- **IPC変更時のテスト**: コマンド/イベントを追加・変更したら、対応するテストを必ず追加する

#### 将来的な強化

- `tauri-specta` 等による型の自動生成で、文書ではなくコードが契約になる状態を目指す
- `dependency-cruiser` / `eslint-plugin-boundaries` によるimport制約の自動チェック

**関連ルール**: `string-contract`, `ipc-test-contract`

### P3: エラーは境界ごとに型を持つ

**各モジュール境界には専用のエラー型を定義する。`String` でのエラー伝播を避け、呼び出し元がエラー種別を判別できるようにする。**

現状は `Result<T, String>` と `.map_err(|e| e.to_string())` が多用されている。
これでは呼び出し元がエラーの種別（PTYが見つからない / IO失敗 / ロック失敗）を判別できず、
ユーザーへの適切なフィードバックやリトライ判断が不可能になる。

#### 方針

**Rust Domain層**: `thiserror` でモジュール単位のエラー型を定義する。

```rust
// ✅ エラー種別が判別可能
#[derive(Debug, thiserror::Error)]
pub enum PtyError {
    #[error("PTY {0} not found")]
    NotFound(PtyId),
    #[error("Failed to open PTY: {0}")]
    OpenFailed(#[source] anyhow::Error),
    #[error("Failed to spawn command: {0}")]
    SpawnFailed(#[source] anyhow::Error),
    #[error("IO error: {0}")]
    Io(#[from] std::io::Error),
}

// ❌ 文字列で伝播（種別判別不可）
fn write(&self, id: PtyId, data: &[u8]) -> Result<(), String> {
    // ...
    .map_err(|e| format!("Failed to write to PTY: {e}"))
}
```

**IPC境界（Commands層）**: Domain のエラー型を Tauri 互換のシリアライズ可能な形式に変換する。

**フロントエンド**: IPC経由で受け取ったエラーを種別に応じて表示する（将来的に）。

#### 適用タイミング

新しいモジュールやコマンドを追加する際はこの原則に従う。
既存コードの `String` エラーは、モジュールに手を入れる機会に段階的に移行する。

### P4: OSリソースは生成と解放をペアで設計する

**PTY、子プロセス、リーダースレッド等のOSリソースは、生成するコードに必ずクリーンアップパスを含める。**

ターミナルアプリはタブの開閉のたびにOSリソース（プロセス、スレッド、ファイルディスクリプタ）を生成する。
リークはサイレントに進行し、長時間使用でメモリ消費の累積やプロセスの残留を引き起こす。

#### 守るべきこと

- `spawn_command` の戻り値（`Child`）を捨てない — プロセス終了検知に必要
- リーダースレッド終了時は対応リソースをマップから削除し、フロントエンドに通知する
- 新しいOSリソースを導入する際は、「このリソースはいつ・どのように解放されるか？」を設計段階で決める

#### 検証方法

- テストでPTYの生成→使用→クローズのフルライフサイクルを検証する（`full_lifecycle` テスト）
- `close` 後の操作がエラーを返すことを確認する（`close_then_resize_returns_error` テスト）

**関連ルール**: `resource-lifecycle`

---

### P5: 安全装置を緩めず、単純さを保つ

**lint・型チェック・テストの厳格さを維持する。新しいライブラリや抽象化の導入前に、既存の手段で解決できないか検討する。**

この原則は2つの側面を持つ:

#### 安全装置を緩めない

- clippy、ESLint、TypeScript strict オプションのエラーが出たら、**設定を変更するのではなくコードを修正する**
- `#[allow(...)]`、`// eslint-disable`、`@ts-ignore` は最終手段。使う場合は理由をコメントに書く
- テストをスキップする `#[ignore]` や `.skip()` は一時的なものとして扱い、恒久化しない

#### 単純さを保つ

- このプロジェクトの本質的な複雑さは **PTY/OS統合** にある。UI側の複雑さは最小に保つ
- 新しい依存ライブラリの導入前に: 「標準ライブラリ / 既存の依存で解決できないか？」を検討する
- 早すぎる抽象化を避ける: 同じパターンが3回現れるまでは、ヘルパーや共通化を急がない

**関連ルール**: `config-protection`
